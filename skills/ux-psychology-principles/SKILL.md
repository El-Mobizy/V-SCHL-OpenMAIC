# UX Psychology Principles

Apply these six principles to every interface you design, critique, or write copy for. They are not optional garnish — they are the default lens. When producing a design, mockup, flow, or review, explicitly check each principle against the work.

**Ethical guardrail:** Use these to reduce friction and communicate value honestly. Never use them to deceive, fabricate urgency or losses that aren't real, or trap users. Loss framing must describe true consequences; anchors must be real prices.

---

## 1. Smart Defaults — Overcome Decision Fatigue

**Problem:** Every blank field is a decision. Stack enough empty boxes (e.g., a complex booking form) and users abandon the flow.

**Principle:** Pre-fill fields with the most common or most likely choice. Roughly 70–90% of users keep defaults because they read them as a trusted recommendation.

**Apply it:**
- Default every form field where a sensible majority choice exists (dates, quantities, plans, locations, notification settings).
- Reframe the user's job from "fill out from scratch" to "scan and adjust what doesn't fit."
- Derive defaults from context when possible (detected location, previous behavior, most popular option).

**Check:** Are there blank fields that could carry an intelligent default? If yes, fill them.

---

## 2. The Goal Gradient Effect — Manufacture Momentum

**Problem:** The closer people feel to finishing, the faster they move. Starting at 0% is deflating and stalls momentum.

**Principle:** Never start a user at zero. Count an action they've already taken (creating the account, opening the app) as the first completed step.

**Apply it:**
- Start onboarding progress bars at ~20%, crediting account creation as step one.
- Break tasks into steps so early wins come fast.
- Show visible, advancing progress everywhere completion matters (profiles, setup, checklists).

**Evidence:** A 10-slot loyalty card with 2 slots pre-stamped gets completed at nearly double the rate of an empty 8-slot card — identical effort, different feeling.

**Check:** Does any progress indicator start at 0%? If yes, give the user a head start they've already earned.

---

## 3. Reciprocity — Give Before You Take

**Problem:** Demanding an email or account before showing any value is like a restaurant asking for your credit card before showing the menu. People leave.

**Principle:** Deliver something genuinely useful *before* asking for commitment. A free gift creates an unconscious pull to return the favor.

**Apply it:**
- Show real partial results before the signup gate: the user's actual score, top issues found, a preview of the analysis — not a blurred screenshot behind a lock icon.
- Position signup as the way to *save or extend* value already received ("Sign up to keep your full report"), not the toll to see anything at all.
- Let users experience the core loop of the product first.

**Check:** Does the flow ask for email/account before delivering any real value? If yes, reorder it.

---

## 4. The IKEA Effect — Build Ownership Before Registration

**Problem:** People value what they helped build far more than an identical thing handed to them. Ownership makes it painful to walk away.

**Principle:** Let users build, customize, or invest effort *before* the registration screen.

**Apply it:**
- Put meaningful customization ahead of signup: name their workspace, pick a color palette, set goals, complete a first lesson (Duolingo's pattern).
- Make the registration button feel like *preserving their work* — closing the tab should feel like throwing away something they made.
- Keep the pre-signup building phase short but genuinely personal.

**Check:** What has the user invested before you ask them to register? If the answer is "nothing," add a building step.

---

## 5. Loss Aversion — Reframe the Stakes

**Problem:** Gain-framed pitches ("Upgrade for more storage!") are weak. The pain of losing something is roughly twice as strong as the pleasure of gaining it.

**Principle:** Highlight what the user stands to *lose* by doing nothing, activating status-quo bias in your favor.

**Apply it:**
- Show the concrete things at risk — the actual files, streaks, data, or progress — not abstract benefits.
- Add real deadlines or countdowns when they truthfully exist.
- Make the dismiss action acknowledge the loss: "I'll risk it" instead of a frictionless "Maybe Later."

**Guardrail:** Only frame losses that are real. Fabricated scarcity or fake deletion threats are dark patterns — don't do it.

**Check:** Is the CTA framed purely around gains? If yes, reframe around what inaction costs.

---

## 6. The Contrast Effect — Anchor Value

**Problem:** The brain never judges a price in a vacuum; it judges it against whatever number it just saw.

**Principle:** Never present a cost in isolation. The first number the user sees becomes the ruler for everything after it.

**Apply it:**
- Place add-ons and upsells next to a larger relevant number: a $50/month protection plan feels expensive alone ($600/year), but beneath a $1,900 laptop labeled "just 2.6%" it feels like a rounding error.
- Sequence pricing pages so the anchor (highest tier, original price, competitor cost) appears first.
- Express small costs as a percentage or fraction of a bigger, already-accepted amount.

**Check:** Is any price shown alone? If yes, give it an honest anchor.

---

## Design Review Checklist

Before delivering any UX design, flow, or copy, verify:

1. **Defaults** — Every field that can be pre-filled, is.
2. **Momentum** — No progress indicator starts at 0%; early steps are credited.
3. **Reciprocity** — Real value is delivered before any signup/email ask.
4. **Ownership** — The user builds or customizes something before registration.
5. **Loss framing** — Key CTAs show what inaction costs (truthfully), and dismissals acknowledge the trade-off.
6. **Anchoring** — No price appears in isolation; every cost has a relative frame.

If a principle doesn't apply to the current screen, note why. Otherwise, apply it.
